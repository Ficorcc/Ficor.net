# vii.ink-fadmin 管理后台实施方案

在 `vii.ink/` 同级新建独立项目 `vii.ink-fadmin/`，用 SvelteKit + Cloudflare Workers 实现。部署到独立子域名 `fadmin.vii.ink`，应用内 base path 配置为 `/fadmin`（满足"地址为 /fadmin"的要求，完整访问地址 `fadmin.vii.ink/fadmin`）。视觉风格完整复刻主站"暖白纸感 + 中国红 + 宋体/楷体"设计系统。

---

## 一、技术栈

| 层 | 选型 |
|---|---|
| 框架 | SvelteKit + `@sveltejs/adapter-cloudflare`（部署 Workers） |
| 数据库 | Cloudflare D1（评论、会话、定时任务、审计、限流计数、配置、访问统计） |
| 存储 | Cloudflare R2（Markdown 文章、上传图片、每日备份包） |
| AI | Cloudflare Workers AI（默认通义千问，可切 Llama / DeepSeek） |
| 图片处理 | Photon WASM（WebP 转换 + 水印） |
| 部署触发 | GitHub Actions `workflow_dispatch` API |
| Schema 校验 | Zod（复刻主站 `content.config.ts` 的 essay/bits/memo schema 与 slug 规则） |

## 三、核心架构

**1. 单一 API 入口 + event 分发** — 所有写操作走 `/fadmin/api/[...event]`，请求体带 `event` 字段，由 `dispatcher.ts` 路由到 handler。事件枚举：

| event | 说明 |
|---|---|
| `AUTH_LOGIN` / `AUTH_LOGOUT` | 登录 / 登出 |
| `CSRF_ISSUE` | 颁发 CSRF token（绑定会话） |
| `COMMENT_SUBMIT` / `COMMENT_MODERATE` / `COMMENT_DELETE` | 评论提交（公开，走限流）/ 审核 / 删除 |
| `CONTENT_LIST` / `CONTENT_GET` / `CONTENT_SAVE` / `CONTENT_DELETE` / `CONTENT_SEARCH` | 文章 CRUD + 全文搜索 |
| `CONTENT_PUBLISH` | 手动触发主站部署 |
| `IMAGE_LIST` / `IMAGE_UPLOAD` / `IMAGE_DELETE` | 图片管理 |
| `AI_POLISH` / `AI_METADATA` / `AI_MODERATE` | 润色（流式）/ 补 frontmatter / 评论审核 |
| `SCHEDULE_CREATE` / `SCHEDULE_CANCEL` | 定时发布 |
| `CONFIG_GET` / `CONFIG_UPDATE` | 系统设置 |
| `AUDIT_LIST` | 审计日志查询 |
| `HEALTH_CHECK` | 健康检查 |

`hooks.server.ts` 串行执行：鉴权（除 login/COMMENT_SUBMIT 外都要会话）→ CSRF 校验（所有 mutation）→ 限流（D1 计数）→ handler 执行 → 审计记录（敏感操作）。

**2. 鉴权链**：密码登录用 WebCrypto PBKDF2 比对（密码哈希存 config 表或环境变量）→ 生成 UUID token 写 `sessions` 表 + 设 httpOnly/Secure/SameSite=Lax cookie，7 天过期，滑动续期。CSRF token 绑定会话，从 `CSRF_ISSUE` 取，前端 mutation 请求头携带，服务端比对后失效或滑动。

**3. 内容存储**：文章以 `essay/<slug>.md`、`bits/<timestamp>.md`、`memo/index.md` 存 R2。frontmatter 用 YAML，schema 用 Zod 复刻主站定义（含 slug kebab-case 校验、保留 slug 黑名单、bits 图片宽高校验）。保存成功后调用 GitHub `workflow_dispatch` 触发主站仓库 Actions 重建。全文搜索：遍历 R2 `content/` 前缀列表 → 读取正文 → 关键词匹配 → 返回标题+摘要片段。

**4. AI 能力**：
- **润色**：Workers AI streaming（`ai.run(model, { stream: true })`），ReadableStream 逐 token 通过 SSE 推前端，前端实时渲染。
- **元数据补全**：读正文 → AI 返回 JSON（title/description/date/tags）→ 填入 frontmatter 表单供确认。
- **评论审核**：对新评论做 spam/敏感判定，返回 approve/review/spam 建议。
- 模型名存 config 表，默认 `@cf/qwen/qwen1.5-14b-chat-aliyun`，支持切换。

**5. 定时发布**：wrangler cron `*/15 * * * *` → `scheduled()` 扫 `schedules` 表到期项，用 D1 行级乐观锁（`UPDATE ... SET status='running' WHERE id=? AND status='pending' RETURNING`）防并发，到期触发部署并标记完成。

**6. 数据备份**：wrangler cron `0 19 * * *`（UTC 19点 = 北京 03:00）→ 导出 D1 各表为 JSON → 打包存 R2 `backups/YYYY-MM-DD.tar.gz`，清理 30 天前，全程 best-effort（失败仅记审计 warn）。

**7. 限流**：`ratelimit` 表按 IP+时间窗口计数，跨实例生效，超阈值返回 429。

**8. 图片处理**：上传走 R2。可选 WebP 转换 + 水印（Photon WASM），水印参数（水印图/位置 9 宫格/大小比例）存 config。图片列表读 R2 `images/` 前缀。

**9. 评论管理**：主站评论用 Waline。fadmin 的评论管理通过 Waline Open API 读取列表、审核、删除（server 端代理，避免暴露 Waline token 到前端）。AI 审核作为辅助标记。不改动主站前端。

## 四、设计系统复刻

搬运主站全套 token 到 `app.css`：
- **色板**：`--bg:#fffefc`（暖白纸感）/ stone 暖灰（`--text:#292524` `--muted:#57534e` `--faint:#a8a29e` `--border:#e7e5e4`）/ `--accent:#b91c1c`（中国红）；暗色 `data-theme="dark"` 覆盖为 `#1a1a1a/#e5e5e5/#f87171`。
- **字体**：正文 Noto Serif SC（宋体）/ 辅助 LXGW WenKai Lite（楷体，label/hint/meta）/ 代码 Sarasa Mono SC。本地 woff2 或 CDN。
- **圆角**：9/10/14/18px + 999px（徽章）。
- **卡片**：hover 显 `--panel` 底，14px 圆角，无常驻边框；分组面板 18px + 柔光阴影 `0 20px 40px -34px rgba(0,0,0,.35)`。
- **控件**：输入框楷体 14px + `box-shadow:0 0 0 3px var(--panel)` focus 光晕；按钮 `--panel` 底/实心黑（主操作）/红字（危险）；分段控件浮雕（内嵌高光）。
- **布局**：侧栏 + 主区域 shell，复刻主站侧栏导航交互（图标按钮 hover scale、主题切换旋转）。
- **暗色模式**：`data-theme` 属性方案 + 内联防闪烁脚本（复刻 `BaseLayout.astro:55`）。

## 五、D1 表结构（migrations/0001_init.sql）

`sessions(id, token, ip, ua, created_at, expires_at)` / `comments(id, post_id, author, email, link, content, status, source, created_at, ai_verdict)` / `config(key PK, value JSON, updated_at, revision)` / `schedules(id, collection, slug, scheduled_at, status, deploy_triggered, created_at)` / `audit(id, session_id, action, target, detail JSON, ip, created_at)` / `ratelimit(ip, window_start, count)` / `stats(key, value, updated_at)`。

config 表预置：`site`（标题/描述/作者）、`ai`（model/prompt 模板）、`ratelimit`（阈值/窗口）、`watermark`（启用/图片/位置/大小）、`deploy`（github repo/workflow）。

## 六、构建顺序（TodoWrite 追踪）

**阶段 A — 基础设施**：项目骨架、wrangler/vite/svelte 配置、app.css 设计 token、app.d.ts 类型、D1 migration、hooks 中间件、鉴权（登录页+session+CSRF）、+layout 守卫、侧栏 shell、文档。

**阶段 B — API 层 + 核心页面**：event dispatcher + Zod schemas、R2 content 读写、frontmatter YAML 工具、CONTENT_* handler、GitHub 部署触发、仪表盘、文章列表 + 编辑器（Markdown 编辑器 + frontmatter 表单 + 拖拽上传）。

**阶段 C — AI / 图片 / 搜索**：AI client + 润色（SSE 流式）+ 元数据补全 + 评论审核、Photon WebP/水印、IMAGE_* handler、全文搜索、图片库页面。

**阶段 D — 运维功能**：定时任务（cron scheduled + 并发锁 + 页面）、审计日志（页面 + 筛选）、系统设置（页面 + CONFIG_UPDATE）、健康检查页面、评论管理页面（Waline 代理）、限流、数据备份（cron + R2）。

## 七、交付说明

- 所有 Cloudflare 绑定（D1 database_id、R2 bucket、AI）和密钥用占位符，`wrangler.toml` + `.dev.vars.example` + `docs/SETUP.md` 写清创建步骤，你创建资源后填入即可。
- D1 schema 完整，可 `wrangler d1 migrations apply` 跑通。
- `npm run dev` 可启动；鉴权/CRUD/AI/搜索/定时/备份逻辑完整可读。
- 因体量大（预计数千行），严格按 A→B→C→D 阶段推进，每阶段结束可阶段性验证。
